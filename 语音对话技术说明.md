# xiaoxin-app 语音对话技术说明

> 本文档详细描述 xiaoxin-app 的语音对话机制，包括设备状态机、音频处理流程、通信协议时序以及防回音 VAD 控制策略。

---

## 一、设备状态机

### 1.1 状态定义

```dart
enum DeviceState {
  unknown,      // 未知状态
  idle,         // 空闲状态（热词监听中）
  connecting,   // 正在建立 WebSocket 连接
  listening,    // 聆听中（录音并上传）
  speaking,     // 回复中（播放 TTS 音频）
  configuring,  // 配置中
  error,        // 错误状态
}
```

                             ▼                                │

            ┌──────►│    Listening    │───────────────────────┤
            │       │   (录音上传中)   │   手动停止/超时          │

### 1.3 状态变更触发条件

| 当前状态                 | 触发事件                    | 目标状态     | 处理逻辑                         |
| ------------------------ | --------------------------- | ------------ | -------------------------------- |
| `idle`                   | 热词唤醒 / 手动点击麦克风   | `connecting` | 建立 WebSocket 连接              |
| `connecting`             | 握手成功 (Hello 完成)       | `listening`  | 启用 VAD，等待用户说话           |
| `connecting`             | 连接失败                    | `error`      | 显示错误，可重试                 |
| `listening`              | 收到 `tts start`            | `speaking`   | 禁用 VAD，开始播放               |
| `listening`              | 手动停止 / 超时             | `idle`       | 发送 `listen stop`，断开连接     |
| `speaking`               | 收到 `tts stop` + 播放完成  | `listening`  | 等待播放完成，启用 VAD           |
| `speaking`               | 热词唤醒 / 手动打断         | `listening`  | 中断播放，发送 `abort`，启用 VAD |
| `speaking`               | 手动停止                    | `idle`       | 中断播放，断开连接               |
| `listening` / `speaking` | WebSocket 断开 (服务端超时) | `idle`       | 清空状态，恢复待机               |

### 3.2 录音 → 发送流程

```dart
// AudioProcessor._onRecordData()
void _onRecordData(Uint8List pcmData) {
  // 1. 条件写入预缓存（用于 VAD 起始补偿）
  //    只有在 VAD 启用或正在说话时才写入，避免 TTS 回声被录入
  final shouldWritePreCache = _vadEnabled || _wasSpeaking;
  if (shouldWritePreCache) {
    _preCache.write(pcmData);
  }

  // 2. 始终进行 KWS 检测（支持热词打断）
  //    在 TTS 播放时，VAD 被 disable，但 KWS 仍然活跃
  _processVadKws(pcmData);

  // 3. 获取 VAD 检测结果
  final isSpeaking = VoiceEngine.instance.isSpeaking;

  // 4. 检测语音开始边缘
  if (isSpeaking && !_wasSpeaking) {
    // 【重要】通知服务端开始监听，初始化服务端 VAD
    ProtocolService.instance.sendStartListening(mode: 'auto');

    // 发送预缓存数据（补偿语音开头）
    final preCacheData = _preCache.readAll();
    _pcmBuffer.insertAll(0, preCacheData);
  }

  // 5. 检测语音结束边缘
  if (!isSpeaking && _wasSpeaking) {
    // 发送剩余缓冲区数据
    _flushBuffer();

    // 通知服务端语音结束
    ProtocolService.instance.sendStopListening();
  }

  _wasSpeaking = isSpeaking;

  // 6.【关键】只有 VAD 检测到人声时才发送音频
  if (isSpeaking) {
    _pcmBuffer.addAll(pcmData);

    // 7. 编码并发送
    while (_pcmBuffer.length >= _bytesPerFrame) {
      _encodeAndSend();
    }
  }
}
```

> **预缓存写入策略**：只在 `_vadEnabled || _wasSpeaking` 时写入预缓存。这避免了：
> 
> - **Idle 阶段**：无会话，预缓存音频无用
> - **Speaking 阶段**：TTS 回声会污染预缓存

### 3.3 接收 → 播放流程（直接 PCM 播放）

```dart
// AudioPlayerService.playOpusData()
Future<void> playOpusData(Uint8List opusData) async {
  // 1. Opus 解码 → PCM
  final pcmData = OpusService.instance.decode(opusData);
  if (pcmData == null) return;

  // 2. 将 PCM 数据直接推送到音频输出 (flutter_pcm_sound)
  final int16List = pcmData.buffer.asInt16List();
  await FlutterPcmSound.feed(PcmArrayInt16.fromList(int16List.toList()));
}
```

> **简化说明**：相比之前的 WAV 转换 + 播放队列方案，现在采用 **直接 PCM 播放**，无需创建临时文件或维护播放列表，延迟更低、实现更简单。

### 3.4 播放延迟优化：按需重建机制

为解决 Android 端 `AudioTrack` 频繁重建导致的 1.5~2秒 延迟，实现了**条件重建机制**：

1. **正常轮次**：当 `_feedCount == 0`（即上一轮音频已完全播完，缓冲区为空）时，收到 `tts start` **不重建播放器**，直接复用当前实例。此时延迟接近 0。
2. **打断场景**：当 `_feedCount > 0`（即缓冲区有残留数据）时，必须执行 `stopPlayback`（包含 `release` + `setup`）以清空缓冲区。

这种机制在保证打断逻辑正确的前提下，消除了 90% 场景下的客户端播放延迟。

## 四、KWS/VAD 控制机制

### 4.1 常驻热词唤醒架构

xiaoxin-app 采用**常驻热词唤醒**策略：KWS 在应用启动后持续运行，不受会话状态影响。

#### 状态与音频处理矩阵

| 状态           | 录音 | KWS | VAD | 发送音频 | 播放 |
| -------------- | ---- | --- | --- | -------- | ---- |
| **idle**       | ✅    | ✅   | ❌   | ❌        | ❌    |
| **connecting** | ✅    | ✅   | ❌   | ❌        | ❌    |
| **listening**  | ✅    | ✅   | ✅   | ✅        | ❌    |
| **speaking**   | ✅    | ✅   | ❌   | ❌        | ✅    |

#### 唤醒流程（按状态区分）

```
应用启动 → 初始化 → 开始录音 → 启用 KWS → idle (等待唤醒)
                                     │
       ┌─────────────────────────────┘
       │ 热词检测到 "小新小新"
       ▼
   ┌──────────────────────────────────────────────┐
   │              _onKeywordDetected()            │
   └──────────────────────────────────────────────┘
                  │        │        │
        idle      │ listening       │ speaking
                  ▼        ▼        ▼
       ┌──────────┐  ┌─────────┐ ┌───────────┐
       │ 建立连接   │ │ 忽略     │ │ 打断播放    │
       │ 发送detect│ │ VAD已在  │ │ 发送detect │
       │ 发送预缓存 │ │ 工作中    │ │ 清空缓冲   │
       └──────────┘ └──────────┘ └───────────┘
              │          │           │
              ▼          ▼           ▼
          listening   (继续)     listening
```

#### 状态分支处理说明

|     状态      | 发送 detected | 发送预缓存 | 启用 VAD | 清缓冲 | 说明                                         |
| :-----------: | :-----------: | :--------: | :------: | :----: | -------------------------------------------- |
|   **idle**    |     ✅ 是      |    ✅ 是    |    -     |   -    | 建立新会话，发送热词+预缓存                  |
| **listening** |     ❌ 否      |    ❌ 否    |  ✅ 确保  |   -    | VAD 已在工作，让 ASR 完整识别"热词+后续命令" |
| **speaking**  |     ✅ 是      |    ❌ 否    |   ✅ 是   |  ✅ 是  | 打断播放，清空回音缓存                       |

> **优化原因**：
> 
> - **listening 不发送 detected**：VAD 已在工作，楼续发送音频到服务端，发送 detected 会导致服务端把热词单独处理，截断用户后续输入
> - **speaking 不发送预缓存**：播放期间预缓存写入的是播放回音，发送会导致 ASR 误识别
> - **speaking 清空缓冲**：避免回音污染后续录音

### 4.2 与 xiaozhi-esp32 的差异

| 特性         | xiaozhi-esp32            | xiaoxin-app                     |
| ------------ | ------------------------ | ------------------------------- |
| 音频发送策略 | **持续发送**，服务端 VAD | **按需发送**，客户端 VAD        |
| 回音消除     | 硬件 AEC / 服务端 AEC    | **播放完成后启用 VAD**          |
| VAD 检测位置 | 服务端 (VadService)      | 客户端 (VoiceEngine)            |
| 预缓存       | 无需                     | **需要**（补偿 VAD 起始延迟）   |
| listen start | 会话开始时发送一次       | **每次语音开始时发送**          |
| listen stop  | 服务端 VAD 检测结束      | **客户端 VAD 检测到语音结束时** |
| KWS 启动时机 | 上电后立即               | 应用初始化完成后立即            |

### 4.3 VAD 启用策略（播放完成后启用）

> **核心理解**：收到 `tts stop` 后，需要等待音频缓冲区**播放完毕**再启用 VAD，避免回音。

#### 实现机制

```dart
// SessionManager._handleTtsMessage
else if (message.isStop) {
  // TTS 结束，等待播放完成后再启用 VAD
  _isSpeaking = false;
  _setState(DeviceState.listening);

  // 标记流结束，等待播放完成
  AudioPlayerService.instance.markStreamEnd(() {
    // 【重要】只清空发送缓冲区，保留预缓存
    // 预缓存需要保留，用于 VAD 检测到语音开始时补偿语音开头
    AudioProcessor.instance.clearSendBuffer();
    // 启用 VAD
    AudioService.instance.enableVad(true);
  });
}
```

#### 播放完成检测（AudioPlayerService）

```dart
// 使用 flutter_pcm_sound 的 feedCallback 检测播放完成
FlutterPcmSound.setFeedCallback(_onFeedCallback);

void _onFeedCallback(int remainingSamples) {
  // 当缓冲区为空且已标记流结束时，触发完成回调
  if (_waitingForPlaybackComplete && remainingSamples == 0) {
    _triggerPlaybackComplete();
  }
}

void markStreamEnd(OnPlaybackCompleteCallback onComplete) {
  _waitingForPlaybackComplete = true;
  _onPlaybackComplete = onComplete;

  // 如果没有音频被 feed，直接触发完成
  if (_feedCount == 0) {
    Future.delayed(const Duration(milliseconds: 100), _triggerPlaybackComplete);
    return;
  }

  // 兜底：2秒后强制触发（防止 feedCallback 不工作）
  _fallbackTimer = Timer(const Duration(seconds: 2), () {
    if (_waitingForPlaybackComplete) {
      _triggerPlaybackComplete();
    }
  });
}
```

> **增强特性**：代码增加了 `_feedCount` 计数和 `_fallbackTimer` 兜底机制，处理无音频和回调不触发的边缘情况。

#### 时序场景

```
服务端发送最后一帧音频 → 发送 tts stop → 客户端收到
                                          ↓
               播放器缓冲区可能还有 200-500ms 音频没播完
                                          ↓
               feedCallback 检测到 remainingSamples == 0
                                          ↓
               触发 onPlaybackComplete → 清空发送缓冲区 → 启用 VAD
```

> **注意**：播放完成后只清空**发送缓冲区**（`clearSendBuffer`），**保留预缓存**以补偿后续语音开头。

### 4.4 预缓存机制 (CircularAudioBuffer)

```
┌─────────────────────────────────────────────────────────────┐
│                    预缓存环形缓冲区                            │
│                                                             │
│  ┌────┬────┬────┬────┬────┬────┬────┬────┐                  │
│  │ -4 │ -3 │ -2 │ -1 │  0 │  1 │  2 │  3 │ ← 帧序号          │
│  └────┴────┴────┴────┴────┴────┴────┴────┘                  │
│                         ↑                                   │
│                    VAD 检测到语音起始                         │
│                                                             │
│  readAll() 时取出 [-4, -3, -2, -1] 帧，补偿语音开头             │
└─────────────────────────────────────────────────────────────┘
```

**作用**：VAD 检测存在滞后性，检测到"语音开始"时，实际语音已经开始了几帧。预缓存保留这些帧，在 VAD 起始边缘插入发送缓冲，避免语音被截头。

### 4.5 VAD 重置机制

> **重要**：在 `enableVad(true)` 时，只有从"禁用→启用"才重置 `_wasSpeaking = false`，防止打断正在进行的语音检测。

```dart
Future<void> enableVad(bool enable) async {
  final wasEnabled = _vadEnabled;
  _vadEnabled = enable;
  VoiceEngine.instance.enableVad(enable);

  // 只在"禁用→启用"时重置状态
  // 如果 VAD 已经在工作（如 listening 状态下热词检测），不重置 _wasSpeaking
  if (enable && !wasEnabled) {
    _wasSpeaking = false;  // 从禁用变为启用时重置
    await RecordService.instance.resumeRecording();
  }
}
```

> **注意**：`enableVad()` **不再清空预缓存**。预缓存的清空由以下方法控制：
> 
> - `clearBuffers()`：清空发送缓冲区 + 预缓存（热词打断播放时调用）
> - `clearSendBuffer()`：只清空发送缓冲区，保留预缓存（TTS 播放完成后调用）

## 五、通信协议时序（小智协议）

### 5.1 消息类型

| 类型     | 方向 | 说明                                     |
| -------- | ---- | ---------------------------------------- |
| `hello`  | 双向 | 握手，交换能力和参数                     |
| `listen` | C→S  | 录音状态控制 (start/stop/detect)         |
| `tts`    | S→C  | TTS 状态控制 (start/stop/sentence_start) |
| `stt`    | S→C  | 语音识别结果                             |
| `llm`    | S→C  | LLM 回复文本和情绪                       |
| `abort`  | C→S  | 打断当前播放                             |
| `mcp`    | 双向 | MCP 工具调用                             |
| Binary   | 双向 | Opus 音频数据                            |

### 5.2 listen 消息时序

```
语音开始 (VAD 检测到):
  Client → Server: {"type": "listen", "state": "start", "mode": "auto"}

语音结束 (VAD 检测到):
  Client → Server: {"type": "listen", "state": "stop"}

打断 TTS:
  Client → Server: {"type": "abort", "reason": "wake_word"}
```

> **重要**：与 ESP32 不同，xiaoxin-app 的 `listen start` 在**每次 VAD 检测到语音开始时发送**，
> 而 `listen stop` 在**每次 VAD 检测到语音结束时发送**。这样服务端 VAD 会话能正确初始化和重置。

### 5.3 tts 消息时序

```
TTS 开始:
  Server → Client: {"type": "tts", "state": "start"}

句子开始 (用于 UI 显示):
  Server → Client: {"type": "tts", "state": "sentence_start", "text": "你好..."}

TTS 结束:
  Server → Client: {"type": "tts", "state": "stop"}
```

## 六、音频焦点与并发播放/录音

### 6.1 问题背景

在 Flutter 中，播放器和录音插件各自管理音频焦点，可能导致：

- 播放时请求音频焦点 → 录音暂停 → 热词无法检测
- 断开时恢复录音 → 请求音频焦点 → 告别语被截断

### 6.2 解决方案

#### 1. 配置 AudioSession

```dart
await session.configure(AudioSessionConfiguration(
  // Android: 媒体模式（使用扬声器播放，而非听筒）
  androidAudioAttributes: const AndroidAudioAttributes(
    contentType: AndroidAudioContentType.speech,
    usage: AndroidAudioUsage.media,  // media 模式确保从扬声器播放
    flags: AndroidAudioFlags.none,
  ),
  // iOS: 同时录音和播放，默认使用扬声器
  avAudioSessionCategory: AVAudioSessionCategory.playAndRecord,
  avAudioSessionCategoryOptions:
      AVAudioSessionCategoryOptions.defaultToSpeaker |
      AVAudioSessionCategoryOptions.allowBluetooth,
  avAudioSessionMode: AVAudioSessionMode.spokenAudio,
));
```

> **注意**：使用 `AndroidAudioUsage.media` 而非 `voiceCommunication`，确保音频从**扬声器**播放而非听筒。

#### 2. 使用 flutter_pcm_sound 直接播放

`flutter_pcm_sound` 插件直接推送 PCM 数据到音频输出，不会自动请求音频焦点，
与录音可以并发运行。

## 七、打断和恢复机制

### 7.1 热词打断（按状态区分）

```dart
void _onKeywordDetected(String keyword) {
  switch (_currentState) {
    case DeviceState.idle:
      // 无会话，建立新连接
      _handleWakeWordSessionStart(keyword);  // 发送 detected + 预缓存
      break;

    case DeviceState.listening:
      // 【重要】不发送 detected，不发送预缓存
      // VAD 已在工作，让 ASR 完整识别"热词+后续命令"
      AudioService.instance.enableVad(true);
      break;

    case DeviceState.speaking:
      // 打断播放
      _setState(DeviceState.listening);
      ProtocolService.instance.sendWakeWordDetected(keyword);
      // 【重要】不发送预缓存（含回音），清空缓冲区
      AudioProcessor.instance.clearBuffers();
      AudioService.instance.enableVad(true);
      break;
  }
}
```

#### 优化说明

| 优化点                    | 原问题                                 | 解决方案                      |
| ------------------------- | -------------------------------------- | ----------------------------- |
| listening 不发送 detected | 服务端把热词单独处理，截断用户后续输入 | VAD 已在工作，让 ASR 完整识别 |
| speaking 不发送预缓存     | 播放期间预缓存含回音，导致 ASR 误识别  | 清空缓冲区，不发送旧数据      |
| speaking 清空缓冲         | 回音污染后续录音                       | `clearBuffers()` 清空预缓存   |


### 7.2 Android/iOS 平台差异处理

#### 1. 热词打断播放（iOS 特有逻辑）

iOS 的 `AVAudioSession` 管理较为严格，当 `active recording` (录音中) 时，若 `flutter_pcm_sound` 尝试重置播放器（需修改 Session Category），会导致 `!pri` (not permitted) 错误。

**解决方案**：采用 **完全停止-重启** 策略。

```dart
// AudioService.stopPlayback()
if (Platform.isIOS && isRecording) {
  // 1. 完全停止录音（释放 Audio Unit 锁）
  await stopRecording(); 
  
  // 2. 重置播放器（此时无录音冲突）
  await player.stopPlayback();
  
  // 3. 重新开始录音
  await startRecording();
}
```

> **性能影响**：相比 Android 的直接重置，iOS 上会有约 100-300ms 的额外延迟（硬件热身），但在打断场景下，这个延迟发生在用户说完唤醒词之后、说命令之前，用户无感知。

#### 2. 回音消除 (Forced VPIO)

- **Android**: 使用 SDK 内置或软件回音消除。
- **iOS**:
  - `record` 插件默认开启 `echoCancel` 会强制使用 Voice Processing I/O (VPIO) 单元。
  - **问题**：VPIO 要求同时有 Input 和 Output。若仅录音（Output 为空），VPIO 会报错 `render err: -1`。
  - **策略**：在 iOS 上禁用 `record` 插件的 `echoCancel`，依靠 `AVAudioSessionMode.spokenAudio` 的系统行为或后期处理，优先保证功能稳定性。

### 7.3 KWS 持续运行


| 状态      | 录音 | KWS | VAD | 发送音频 | 说明                   |
| --------- | ---- | --- | --- | -------- | ---------------------- |
| Idle      | ✅    | ✅   | ❌   | ❌        | 录音运行，仅检测热词   |
| Listening | ✅    | ✅   | ✅   | ✅        | 检测语音 + 热词 + 发送 |
| Speaking  | ✅    | ✅   | ❌   | ❌        | 录音运行，热词可打断   |

服务端会在会话长时间无活动时主动结束会话。

### 9.2 常见问题排查

| 问题                   | 排查方向                                                        |
| ---------------------- | --------------------------------------------------------------- |
| 录音无响应             | 检查权限、RecordService 状态、VAD 是否 enable                   |
| 语音发送但服务端无响应 | 检查 `listen start` 是否在语音开始时发送、WebSocket 连接状态    |
| TTS 无声音             | 检查音频解码、flutter_pcm_sound 是否初始化                      |
| 打断不生效             | 检查 KWS 是否持续运行、`abortSpeaking` 是否调用                 |
| 状态卡住               | 检查 `tts stop` 是否收到、播放完成检测是否工作                  |
| 会话自动结束           | 正常情况：服务端 60s 无活动超时，发送告别语后断开               |
| 断开后界面未恢复       | 检查 `_onDisconnected` 是否调用、状态是否正确重置为 Idle        |
| 告别语被截断           | 检查断开时是否清空播放队列（不应清空）                          |
| 播放后热词不工作       | 检查 `AudioSessionService.initialize()` 是否调用                |
| TTS 播放产生回音       | 检查是否等待播放完成后才启用 VAD                                |
| 上一条消息被清空       | 检查 `_updateOrAddAiMessage` 是否正确使用 `_isNewResponse` 标记 |
| 唤醒词识别为"小心"     | 检查 FunASR 版本 >= 1.3.0，服务端热词参数是否正确传递           |

## 十、ASR 热词与后处理

### 10.1 FunASR 版本要求

> **重要**：热词功能需要 `funasr >= 1.3.0`，低版本热词参数会被忽略。

| 依赖       | 最低版本     | 说明                   |
| ---------- | ------------ | ---------------------- |
| **funasr** | **>= 1.3.0** | 热词功能必须使用此版本 |
| Python     | >= 3.10      | 推荐 Conda Python 3.10 |

**升级命令**: `pip install "funasr>=1.3.0"`

### 10.2 热词配置格式

```java
// FunASRSttService.java
// 热词格式：简单列表（FunASR Nano 不支持权重）
private static final String HOTWORDS = "[\"小新小新\",\"小新\"]";
```

> **注意**：FunASR Nano 基于 LLM，热词只是拼接到 Prompt 中，不支持传统 ASR 的权重机制。

### 10.3 服务端 ASR 后处理

Java 服务端在 `DialogueService.processAsrResult()` 中统一处理 ASR 返回结果：

```java
private String processAsrResult(String text) {
    // 1. 短文本过滤：过滤回声/杂音（如 "的。"、"光。"）
    String cleanText = text.replaceAll("[\\p{Punct}\\s]", "");
    if (cleanText.length() <= 1) return null;

    // 2. 唤醒词修正：小[心新]+标点+小[心新] → 小新小新
    String pattern = "小[心新][。，,、.\\s]*小[心新]";
    return text.replaceAll(pattern, "小新小新");
}
```

#### 处理逻辑说明

| 功能       | 问题                               | 解决方案                       |
| ---------- | ---------------------------------- | ------------------------------ |
| 短文本过滤 | 回声/杂音产生如"的。"结果          | 去标点后有效字符 <= 1 则丢弃   |
| 唤醒词修正 | ASR 识别为"小心小心"或"小新。小新" | 正则匹配并统一替换为"小新小新" |
